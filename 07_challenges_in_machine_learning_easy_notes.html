Before Coding: Know the Real Challenges

The teacher says that before moving into coding and building ML models, it is important to understand the practical challenges of Machine Learning.

Most ML problems come from two broad areas:

📊

Data-related challenges

Collection, insufficient data, labels, representation, quality, and useful features.

🤖

Model / production challenges

Overfitting, underfitting, software integration, deployment, monitoring, and cost.

Main message: Making an ML model is not enough. You must also make it useful, reliable, deployable, and affordable.
📥

1Data Collection

Machine Learning is all about learning from data. If you do not have data, there is no point in doing ML.

In courses and college projects, data is usually easy to get. You may download CSV files from Kaggle or receive datasets from teachers. But in real company projects, data gathering is often difficult.

Need DataNo data = no ML
→
Collect DataInternal systems / APIs / scraping
→
Use DataTrain ML model
  • ✓Data may come from an organization’s internal databases.
  • ✓Sometimes you may use web scraping or APIs to collect data.
  • !Collecting data through code can create many technical and legal/practical issues.
  • !If ready-made data is not available, gathering data itself becomes a major project.
🏷️

2Insufficient Data and Labeled Data

Even after collecting data, you may not have enough of it. ML models usually need a good amount of data to perform well.

The video explains that sometimes a weaker algorithm with more data can outperform a stronger algorithm with less data.

📉

Less Data

Even a good algorithm may perform poorly if the training data is too small.

📈

More Data

A simpler algorithm can perform better if it receives much more useful data.

Key idea: Data can be extremely powerful. This is often called the “unreasonable effectiveness of data.”

Labeled data is another problem

For supervised learning, the data also needs labels. For example, if you are building an image classifier, collecting images is not enough. Someone must label each image as cat, dog, car, etc.

  • !Labeling needs human effort.
  • !Labeling can be expensive and time-consuming.
  • !Without labels, many supervised ML tasks cannot be trained properly.
🌍

3Non-Representative Data

Your data should represent the real problem. If your sample is biased or incomplete, the model will learn the wrong pattern.

📌

Sampling Noise

When the sample is too small or incomplete, it may tell only half the story.

Example: fitting a line using only a small part of the full data can produce the wrong model.

⚖️

Sampling Bias

Even a large sample can be biased if the data is collected from the wrong group.

Example: asking only Indian fans who will win the cricket World Cup may give a biased answer.

If the data does not represent the real world properly, the model’s predictions will also be unreliable.

4Poor Quality Data

Data often contains many quality problems. If the input data is bad, the ML model cannot magically fix everything.

🔊

Noise

Random unwanted errors in the data.

❓

Missing Values

Important values may be empty or unavailable.

🚨

Outliers

Extreme values can disturb the model.

🧾

Wrong Format

Different formats make processing difficult.

A common saying in ML: Garbage in, garbage out. Bad data usually creates bad results.

The video mentions that in real ML projects, a large part of the time can go into cleaning and improving data quality. Data cleaning is one of the most important skills for a data scientist or ML engineer.

🧩

5Irrelevant Features and Feature Engineering

Features are the input columns used by the model. If you give useless or irrelevant features, the model may perform badly.

🗑️

Irrelevant Features

Columns that do not help prediction should often be removed.

Example: for predicting marathon participation, location may not directly explain fitness.

🛠️

Feature Engineering

Sometimes we combine existing columns to create a better feature.

Example: height + weight can be used to create BMI.

Choosing which columns to keep, remove, or combine is a major challenge and improves with experience.
🎯

6Overfitting

Overfitting happens when the model memorizes the training data too closely instead of understanding the real pattern.

It performs very well on training data but badly on new unseen data.

Easy analogy: Watching one expensive movie in Gurgaon and concluding “everything in Gurgaon is expensive” is overgeneralizing from limited data.
  • !The model becomes too attached to training points.
  • !It may create a complicated curve that touches almost every training point.
  • !It fails to generalize on new data.
📉

7Underfitting

Underfitting is the opposite of overfitting. It happens when the model is too simple and cannot capture the actual pattern in the data.

  • !The model does not perform well even on training data.
  • !It also performs poorly on new data.
  • ✓The goal is to find a balanced model: not too complex, not too simple.
🔌

8Software Integration

An ML model usually helps users as part of a software product. So after building the model, the next big task is to integrate it into real software.

🌐

Web App

Deploy through a website or backend service.

📱

Mobile App

Use in Android or iOS apps.

🖥️

Other Platforms

Desktop, server, embedded, or special environments.

The teacher points out that ML is still a developing ecosystem. Some languages and platforms do not support ML as smoothly as Python does. Compatibility between libraries and platforms can also be difficult.

Important advice: Do not stop after making a model. Convert it into a product that real users can use.
🚀

9Deployment, Offline Learning, and Online Learning

Deployment means putting the model on a server so users can use it. This is often harder than beginners expect.

📦

Offline / Batch Deployment

The model is trained offline, uploaded to the server, and updated later by retraining and redeploying.

🔄

Online Learning Deployment

The model learns or updates on the server as new data comes. This is powerful but harder to manage.

TrainCreate model
→
DeployRun on server
→
MonitorCheck real-time behavior
→
UpdateRetrain or online update

Cloud providers like AWS, GCP, and Azure provide ML deployment services, but production ML still needs careful monitoring, versioning, data handling, and engineering effort.

Deployment is not just “upload the model.” It includes serving, monitoring, updating, and maintaining the model in production.
💰

10Cost Involved

The teacher says cost becomes clear only when you deploy models at scale. If thousands or lakhs of users use your ML system, hidden costs can become huge.

🖥️

Compute Cost

Training and inference may need powerful servers or GPUs.

🗄️

Storage Cost

Data, models, logs, and backups require storage.

📡

Serving Cost

Every user request may trigger model inference and server usage.

Because of these issues, a dedicated field called MLOps is growing. MLOps deals with deploying, monitoring, maintaining, scaling, and managing ML systems in production.

Final advice from the video: learn how to turn ML models into real software products and deploy them for active users.
🧾

Practical Checklist for Any ML Project

# Challenge Question to Ask What to Do
1 Data Collection Where will the data come from? Use internal data, APIs, scraping, or reliable sources.
2 Insufficient / Labeled Data Do we have enough data and labels? Collect more data, label data, or use suitable techniques.
3 Non-Representative Data Does this data represent the real problem? Avoid sampling noise and sampling bias.
4 Poor Quality Data Is the data clean? Handle noise, missing values, outliers, and formats.
5 Irrelevant Features Which columns are useful? Remove useless features and engineer better ones.
6 Overfitting Is the model memorizing training data? Simplify, regularize, validate, and test on unseen data.
7 Underfitting Is the model too simple? Use better features or a more suitable model.
8 Software Integration Can real users use the model? Integrate into web/mobile/backend products.
9 Deployment Can it run reliably in production? Serve, monitor, update, and maintain the model.
10 Cost Can we afford it at scale? Optimize compute, storage, inference, and operations.

Final Super Simple Summary

  • Data collection is hard: Real-world data is not always available like Kaggle datasets.
  • Data quantity matters: More useful data can beat a better algorithm with less data.
  • Labels matter: Supervised learning needs correctly labeled data.
  • Representation matters: Biased samples create biased models.
  • Quality matters: Noise, missing values, outliers, and wrong formats must be fixed.
  • Features matter: Remove irrelevant columns and create meaningful features.
  • Overfitting: Model memorizes training data and fails on new data.
  • Underfitting: Model is too simple and fails to learn the pattern.
  • Integration and deployment matter: A model must become a usable software product.
  • Cost matters: Production ML has hidden costs, so MLOps is important.
